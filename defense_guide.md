本專案選擇使用 **C++** 實作，以下是與其他語言的效能與開發速度比較：

| 語言 | 效能 | 開發速度 | 內建資料結構 (STL) | 綜合評估 |

## 1. 核心概念與執行流程 (Workflow & Logic)

**目標**：將一個 N×N 的隨機數字拼圖，透過滑動空白格（0），還原成「蝸牛形（Snail）」的螺旋排列。

**程式執行流程**：
1. **輸入與解析**：從檔案讀取盤面或隨機生成確保可解的盤面。
2. **可解性檢查 (Solvability)**：在開始複雜的搜尋之前，先透過數學方法（計算逆序對的奇偶性）判斷目標是否可達。
   - **為什麼這非常必要？**：根據數學定裡，如果你隨便把 9 個數字放進 3x3 的格子裡，剛好會有 **50% 的機率是死局（無解的）**。如果不做可解性數學檢查，A* 會認為可能是路徑還沒找到，從而嘗試把上百萬、上千萬個可能的狀態全部展開，最終會塞爆記憶體並導致程式崩潰 (die a horrible death)。有了數學這道鎖，碰到那 50% 無解的局就能瞬間判定並安全結束。
3. **啟發式搜尋 (A\* Search)**：將起始狀態放入優先隊列（Open List）。
4. **狀態展開**：每次取出優先權最高（$f(n)$ 最小）的狀態，並檢查是否為目標。若不是，則產生其所有合法的鄰居狀態（將 0 上下左右移動），放入隊列中。
5. **回溯路徑**：當找到目標狀態時，透過每個節點記錄的「父節點」指標，一路回溯到起點，形成完整的解題步驟。

## 2. 搜尋演算法 (Search Algorithm & Variant)

**實作版本**：標準的 **A\* (A-Star) Search Algorithm**

**為什麼選擇 A\*？**
A* 是尋找最短路徑最有效的演算法之一，它結合了：
- **Dijkstra 演算法**的特點：考慮已經花費的真實成本 $g(n)$（目前的步數），確保找到的路徑是最佳解。
- **Greedy Best-First Search** 的特點：考慮預估成本 $h(n)$（啟發式函數），引導搜尋方向朝向目標，大幅減少需要探索的狀態數量。

總評分公式為：**$f(n) = g(n) + h(n)$**
每次展開節點時，我們永遠選擇 $f(n)$ 最小的節點。本專案透過改變這個公式，也實作了另外兩種搜尋變體 (Search Variants)：

- **Uniform-Cost Search (UCS)**：將公式改為 **$f(n) = g(n)$**（強制 $h=0$）。這代表我們完全不管離目標還有多遠，只看「目前已經走了幾步」。它像水波一樣一圈一圈往外找，速度非常慢，但**保證絕對能找到步數最少（完美）的解**。
- **Greedy Best-First Search**：將公式改為 **$f(n) = h(n)$**（強制 $g=0$）。這代表我們完全不管已經走了多少冤枉路，只看「眼前這一步評估起來離目標有多近」。它會像貪婪的人一樣筆直衝向終點，**速度非常快、記憶體吃得少**，但缺點是它找到的解**通常不是步數最少的最佳解**。

### 📌 演算法解題能力與極限評估 (Capability Estimation)

在 Defense 時，你必須清楚知道各演算法的數學極限，這能展現你對時間/空間複雜度的深刻理解：

| 演算法 (Search Mode) | 3x3 盤面 | 4x4 盤面 | 5x5 及以上大型盤面 (如 10x10) | 最佳解保證？ | 核心瓶頸 |
|---|---|---|---|---|---|
| **A\*** ($f = g + h$) | ⚡ 瞬間秒殺 | ✅ 可解 (搭配 Linear Conflict 尤佳) | ❌ OOM (記憶體撐爆) / 極慢 | ✅ 保證最短路徑 | 空間複雜度 $O(b^d)$，Open/Closed set 會耗盡 RAM |
| **Uniform-Cost** ($f = g$) | ✅ 可解 (稍慢) | ❌ 極慢 / OOM | ❌ 絕對 OOM | ✅ 保證最短路徑 | 盲目搜尋，完全沒有方向感，展開節點呈幾何級數爆炸 |
| **Greedy** ($f = h$) | ⚡ 瞬間秒殺 | ⚡ 瞬間秒殺 | ✅ **可解 (極速解題)** | ❌ **不保證最短路徑** (可能繞一大圈) | 容易卡在局部最佳解 (Local Optima)，且路徑長度通常很難看 |

> **💡 面試對答重點**：
> 如果被要求跑 10x10 盤面，請大膽切換到 `-a greedy`。因為找出 **10x10 的完美最短路徑** 在普通的 A* 放進電腦記憶體裡是數學上的不可能（狀態數高達 $100! / 2$）。Greedy 雖然會走出繞遠路的解（可能破千步），但它是唯一能在合理時間與空間內降伏巨型盤面的變體。

## 3. 啟發式函數與可容許性 (Heuristics & Admissibility)

專案要求至少實作 3 種 Admissible（可容許的）啟發式函數。
> **什麼是 Admissible？**
> 一個啟發式函數 $h(n)$ 是 Admissible，代表它「**永遠不會高估**」到達目標的真實成本。也就是說，它計算出來的預估步數，一定小於或等於實際需要的步數。這是確保 A* 演算法能找到最佳解的充要條件。

### A. Manhattan Distance (曼哈頓距離) - **必備**
- **計算方式**：盤面上每個數字（除了 0）現在位置到其目標位置的「水平距離 + 垂直距離」總和。
- **Admissible 證明**：因為每次移動只能上下左右走一格（即曼哈頓移動），一個方塊要走到目標的最少步數就是它的曼哈頓距離。既然我們忽略了其他方塊的阻擋，實際需要的步數只會大於或等於曼哈頓距離，因此絕對不會高估。

### B. Hamming Distance (漢明距離)
- **計算方式**：盤面上「不在正確位置上」的方塊總數。
- **Admissible 證明**：如果一個方塊不在正確位置，它至少需要移動 1 步才能歸位。我們假設每個不對的方塊只要 1 步就能飛到目標，這顯然是非常樂觀的估計，實際步數必定 $\ge$ 不對位的方塊數量，因此不會高估。

### C. Linear Conflict (線性衝突) - **效能最強**
- **計算方式**：在 Manhattan Distance 的基礎上，如果兩個方塊都在正確的行（或列）上，但它們的**順序顛倒**了，那麼為了讓它們歸位，其中一個方塊必須先「讓路」（也就是離開目標行/列），等另一個過去後再回來。這表示它們除了曼哈頓距離外，還需要**至少 2 步**額外的移動。
- **Admissible 證明**：我們只對確定會發生「讓路」行為的方塊加上確切的最低額外步數（2步）。由於這是在幾何物理限制下的絕對下限，所以加上這 2 步依然不會高估真實所需的總步數。實測中，Linear Conflict 能大幅減少 4x4 盤面展開的節點數（從數百萬降至數十萬）。

## 4. 資料結構設計與選擇 (Data Structures)

為了讓 A* 能夠在合理時間內處理大型拼圖，資料結構的選擇是效能的關鍵（也就是解決 Time & Space Complexity 的瓶頸）。

1. **盤面表示法 (`std::vector<int>`)**：
   - 使用一維陣列取代二維陣列來儲存 N×N 的數字。
   - **原因**：記憶體連續，快取 (Cache) 命中率高，複製 (Copy) 的速度極快。這在每秒需要產生上千個新狀態的 A* 演算法中是效能核心。

2. **Open List (`std::priority_queue`)**：
   - 負責存放等待探索的節點，並根據 $f(n)$ 自動排序（由小到大）。如果 $f(n)$ 相同，則優先選擇 $h(n)$ 較小的節點（代表離目標更近）。
   - **原因**：Priority Queue 內部是 Min-Heap，推入或取出最小值的時間複雜度為 $O(\log N)$，這遠快於每次去掃描一個 List 或 Array。

3. **Closed Set (`std::unordered_set<Puzzle>`)**：
   - 記錄所有已經被展開（探索過）的盤面狀態。
   - **原因**：A* 必須避免走回頭路。`std::unordered_set` 內部使用 Hash Table，可以在平均 $O(1)$ 的時間內判斷這個盤面前有沒有出現過，確保搜尋極速進行。為了支援這個操作，我們為 `Puzzle` 定義了自訂的 `std::hash` 特化。

4. **路徑回溯圖 (`std::unordered_map<Puzzle, shared_ptr<Node>>`)**：
   - 記錄每個狀態的具體指標（包含 $g$, $h$ 與 `parent`）。
   - **原因**：當我們找到目標狀態時，可以透過不斷呼叫 `node = node->parent` 一路印回最初的起始狀態。使用 `shared_ptr` 能自動管理這些節點佔用的記憶體，避免 Memory Leak。

## 5. 程式碼閱讀順序建議 (Code Reading Guide)

1. **核心數據層**：`Puzzle.hpp` & `Puzzle.cpp`
   - **展示重點**：先了解拼圖在程式裡是怎麼被存的（`std::vector<int>` 及其在 Cache 上的優勢）。講解怎麼生成相鄰節點，以及最重要的「**可解性數學檢查 (`isSolvable`)**」。

2. **資料輸入層**：`Parser.hpp` & `Parser.cpp`
   - **展示重點**：說明如何處理檔案輸入、過濾註解，以及**為什麼隨機生成的盤面一定有解**（因為從 Goal state 往回反推亂走而得）。

3. **評分策略層**：`Heuristics.hpp` & `Heuristics.cpp`
   - **展示重點**：這不僅是 Defense 必考題，也是 A* 的靈魂。展示 Manhattan 與 Linear Conflict，解釋什麼叫做「不會高估成本 (Admissible)」。

4. **搜尋引擎層**：`Solver.hpp` & `Solver.cpp`
   - **展示重點**：這是重頭戲。打開 `Solver::solve()`，展示 `while (!openList.empty())` 迴圈。講解如何用 Priority Queue 挑選最小 $f(n)$，如何用 Unordered Set 避免死胡同，並介紹如何動態切換 A*, Uniform-Cost 和 Greedy。

5. **總指揮中心**：`main.cpp`
   - **展示重點**：最後展示怎麼將命令列的 Parser、Heuristic 函數與 Solver 串接起來，並加入計時器 (`std::chrono`) 列印出完整的執行報告。
